## Interpreting Results

- **Liquidity > $20K**: decent, lower risk of rug
- **Liquidity $5-20K**: thin, higher slippage and risk
- **Liquidity < $5K**: avoid (default filter)
- **Buy/Sell ratio**: >80% buys = FOMO, be cautious
- **Age < 1h**: very early, highest risk
- **Age 1-4h**: discovery phase, good entry if contract is clean
- **Age 4-24h**: established trend, verify before entry
- **Boosts**: paid promotion, not organic — verify independently

## Buy Workflow (IMPORTANT — follow exactly)

After scanning trending tokens, DO NOT run `analyzer.py safety` manually.
It only returns max 45/45 (contract-only) which will ALWAYS get blocked by risk-manager (min 60).

**Correct workflow:**
1. `scanner.py trending --limit 10` → pick candidates with liquidity > $10K
2. For promising tokens, go straight to: `executor.py buy --token <addr> --reason "..."`
3. Executor internally runs full `analyzer.py analyze` (0-100 score) and `risk_manager.py check`
4. If score < 60 or risk check fails → executor blocks the trade automatically

**If you want to preview before buying:**
- Use `analyzer.py analyze <addr>` (NOT `safety`) to see the full 0-100 score
- Then decide whether to call `executor.py buy`

**DO NOT:**
- Run `analyzer.py safety` — it's contract-only (max 45), useless for buy decisions
- Run `risk_manager.py check` manually before `executor.py buy` — executor does this internally
- Pass `--amount` to bypass auto-sizing unless wallet balance is genuinely too low

**NEW: Multi-Token Buy Strategy**
- If `max_positions` is reached (e.g., 5), executor will BLOCK all new buys — even if tokens are high-score.
- Do NOT try to force buys — wait for existing positions to exit.
- Use `executor.py portfolio` to monitor open positions.
- Use `executor.py check-exits` to see pending TP/SL triggers.
- If you need to free up space, manually close a low-performing position with `executor.py sell --id <N> --reason "freeing slot"` before attempting new buys.
- Never ignore the `Open positions >= max` error — it's a budget safeguard, not a bug.

## Contract Check Results (--check-contract)

- ✅ `mint: revoked` — safe, no new tokens can be minted
- ❌ `mint: ACTIVE` — dev can print tokens, HIGH RISK
- ✅ `freeze: revoked` — safe, your tokens can't be frozen
- ❌ `freeze: ACTIVE` — dev can freeze your wallet, HIGH RISK

---

## Pitfalls

- **DEXScreener rate limits**: 300 req/min for pair data, 60 req/min for profiles/boosts.
- **Token profiles are self-submitted**: having a profile doesn't mean legitimacy.
- **Contract check uses Solana RPC**: public RPC has rate limits. Use private RPC for production.
- **Not all scams are detectable**: honeypots, hidden mint via proxy, etc. Always DYOR.
- **pairCreatedAt** is in milliseconds since epoch.

## How to Change SL, TP, Trailing Stop, and Other Config
 
Use `config-propose` to request a change. This logs a proposal that Damian must approve.
 
```bash

python3 ~/.hermes/skills/trade-executor/scripts/executor.py config-propose \
  --key take_profit_pct --value "50" --reason "lower TP for quicker exits"
 
# Change trailing stop from 15% to 10%

  --key trailing_stop_pct --value "10" --reason "tighter trailing for memecoins"
 
# Disable trailing stop (instant sell at TP)

  --key trailing_stop_pct --value "0" --reason "disable trailing, sell at TP"

  --key position_pct --value "3.0" --reason "smaller positions in bear market"
```

### How trailing stop works
When a position hits `take_profit_pct` (e.g. +100%), Guardian does NOT sell immediately.
Instead it activates trailing: if price drops `trailing_stop_pct` (e.g. 15%) from the peak,
GUARDIAN sells. This lets winners run. Set `trailing_stop_pct: 0` to sell instantly at TP.
